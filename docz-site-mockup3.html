Model Context Protocol

Pull any docz repo into your editor.

The reader ships a small MCP server that exposes the read API as callable tools. Connect Claude Code, Cursor, or any MCP-aware client and search, fetch, and cross-reference documents across every registered repo — without leaving your editor. Add a repo by registering its .docz.yaml; the same server then serves it too.

docz-mcp
this server

Exposes the read API as MCP tools — list_docs, get_doc, search_docs, get_links, get_activity — across rfcs, docs, and any repo you register. Read-only; authoring stays in Git.

docz-api
powers this

The static JSON read API the site and MCP both build from — generated from each repo's front matter and the doc-model.hcl type graph. No running service: just files served from the same host.

2Configure your client

Add the server to your MCP client config. The path varies by client; the examples cover the common ones.

// ~/.config/claude-code/mcp.json
{
  "mcpServers": {
    "docz": {
      "command": "docz-mcp",
      "args": ["--api-url", "https://docz.internal/api/v1"],
      "env": { "DOCZ_MCP_TOKEN": "$DOCZ_READ_TOKEN" }
    }
  }
}

3Available tools

Once connected, these tools are available to your LLM. All are read-only and safe to invoke without confirmation.

  • list_docs — list documents with optional repo, type, status, and label filters. Returns id, title, type, status, owner, updated_at.
  • get_doc — fetch one document by id, with rendered body and resolved cross-references.
  • search_docs — full-text search across titles, bodies, and labels in every repo. Ranked, with highlighted snippets.
  • get_links — the reference graph for a document: instantiates, applies, satisfies, decides, and what points back.
  • get_activity — recent changes across all repos, for "what changed this week" prompts.

4Verify it works

Restart your client and ask it about a recent document. It should invoke get_doc and return a coherent answer. A good sanity check:

# in your LLM chat:
"Summarize FW-0001 and list every document that references it."

Fetch a single document by ID

Everything the site and the MCP server render is a static JSON file generated from front matter and the type graph — no running service. IDs are stable across repos.

GET /api/v1/docs/FW-0001
{
  "id": "FW-0001",
  "type": "framework", "repo": "docs",
  "title": "AI Security Framework",
  "status": "draft", "owner": null,
  "links": {
    "applied_by": ["POL-0001"],
    "maps_to": ["OWASP-LLM-2025", "NIST-AI-600-1", "MITRE-ATLAS"]
  },
  "source": "fwsync/frameworks/ai-security.hcl",
  "body_url": "/api/v1/docs/FW-0001.md"
}

List endpoints accept ?repo=, ?type=, ?status=, and ?label= — the same filters the directory and the MCP list_docs tool use.
